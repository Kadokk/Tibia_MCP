# Tibia-MCP — Repository Audit (2026-06-14)

> **What this is:** a principal-level, evidence-based audit of the whole repo as of
> 2026-06-14. Findings cite `file:line`. Use it as the standing record of *where we
> stand*; pair it with [`../README.md`](../README.md) for the short status summary.
> **Grade at time of audit: C+ (bimodal).** The data MCP is solid; the live listener
> is blocked and unproven; the Discord SaaS is clean but hollow scaffolding.

---

## Executive Summary

This repo is two projects at very different maturities welded together.

- The **TibiaData/TibiaWiki/Bazaar data MCP** (sub-project 1) is genuinely well-engineered —
  parameterized SQL throughout, textbook-correct XTEA/RSA crypto, bounds-checked binary
  parsing, graceful stale-cache fallback, 127 tests. On its own: **B+**.
- The **live trade-channel listener** (the actual product differentiator) is **D**: it is
  **blocked from ever connecting to its configured target world** by a no-op BattlEye stub
  (`lib/protocol/src/battleye.cpp:23`), and past that has High-severity frame-decode and
  timeout bugs that would silently drop the very trade messages it exists to capture.
- The **Discord SaaS layer** is disciplined TDD scaffolding but functionally **hollow** —
  every command is a placeholder (`services/discord-bot/src/commands/registry.ts:5-6`) and
  every repository is a type-only stub.

**The defining risk:** a sophisticated multi-language stack is being built on top of a data
pipeline that has **never been proven to capture a single row of live data** (the live smoke
test was deferred — see `docs/listener-smoke-test.md`).

**Top 3 risks:** (1) core data source blocked + unvalidated; (2) the listener violates Tibia
ToS and its perfectly-periodic anti-idle is a bot signature, layering legal + ban exposure
under the monetization plan; (3) no CI / no quality gates across 3 languages, and a live
account credential sat in the working tree (`.env:9-10` — now flagged for rotation).

**Top 3 opportunities:** (1) a ~1-hour live smoke test that validates or kills the pipeline
cheaply; (2) the Bazaar/TibiaData scrapers already produce sellable, **ToS-legal** market data
with no listener at all — the SaaS could ship on that today; (3) small, high-leverage cleanups
(HTTP-status gating, tool-layer de-duplication) remove whole classes of silent failures.

## Repo Map

**Purpose.** A C++ MCP server exposing Tibia data (character/guild/world lookups, item/
creature/spell/quest wiki search, Bazaar scraping, cache mgmt) to LLM clients, plus an
experimental live trade-channel listener that logs into a real account, captures Trade-channel
chat, parses it into structured price offers (regex → Claude Haiku fallback), and stores them
in SQLite. A newer TypeScript Discord bot (`TibiaEdge`) is the intended commercial front-end.

**Stack.** C++17 / CMake 3.20 (FetchContent: nlohmann_json v3.11.3 + googletest v1.14.0;
system CURL + SQLite3; OpenSSL EVP for RSA). TypeScript / Node ESM / discord.js 14 / pg / zod /
vitest for the Discord bot. Playwright (JS) for two login-probe helpers. SQLite shared store;
Postgres planned for the SaaS.

**Architecture — 3 cooperating C++ processes over one SQLite DB + a separate TS service:**

```
                 +------------------------------------------+
   LLM client -->| tibia-mcp (stdio JSON-RPC, 21 tools)     |
                 +-----------------+------------------------ +
                                   | reads
 Live Tibia  --> tibia-listener  --writes raw_messages-->  SQLite (WAL)
 server      (lib/protocol)                               tibia_mcp_cache.db
                                   ^                           ^
                 tibia-parser -----+ regex->Claude->offers ----+
                                                               | (planned) SQLite->Postgres
                              services/discord-bot (TS)  <------+  [scaffolding only]
```

**Key directories.**

| Path | Role |
|---|---|
| `src/mcp/` | MCP server, stdio transport, 21 tool handlers (`src/mcp/tools/*`) |
| `src/sources/` | TibiaData (JSON API) + TibiaWiki + Bazaar (regex HTML scraping) |
| `src/cache/`, `src/store/` | SQLite cache (per-tool TTL) + `TradeStore` (raw_messages/trade_offers/item_registry) |
| `src/parser/`, `src/llm/` | regex parser + LLM parser + item registry; Claude client |
| `src/listener/` | login + channel-join + anti-idle + message sink (`tibia-listener`) |
| `lib/protocol/` | standalone Tibia binary-protocol static lib (crypto, framing, login, opcodes, BattlEye stub) |
| `services/discord-bot/` | TypeScript SaaS layer (commands, services, repositories, formatters, migration SQL) |
| `docs/superpowers/` | 4 design specs + 4 plans |
| `tools/`, `data/items.json` | Playwright probes; ~50 curated items + slang |

**Surprises (facts).** Root `package.json:12` test script is `exit 1`; no CI; live `-wal`/`-shm`
files present (DB has been run, but not proven via *live* capture); a real account password in
cleartext at `.env:10` (gitignored, absent from history — verified); the Discord bot runtime
does nothing despite high-quality tests.

- Parameterized SQL everywhere — zero injection surface.

## Improvement Strategy

Five themes explain ~90% of findings.

1. **The foundation is unvalidated and currently blocked** (S1, A1, T1, P1, P2, S2). Validate the
   riskiest assumption before building on it — run the live smoke test, or pivot the SaaS to the
   **ToS-legal Bazaar + TibiaData scrapers** that already work.
2. **Error handling is inconsistent and occasionally unsafe** (Q1, Q2, Q3, Q6, P3, P4, O3, O4).
   Fail loudly and locally; never cache or silently drop on error. `tibiadata.cpp` already shows
   the right model — propagate it.
3. **No enforced quality gates across a polyglot repo** (O1, O2, T2, T4). CI that builds + tests
   C++ and runs vitest + tsc + eslint, blocking merge; a real migration runner.
4. **The SaaS layer is hollow with latent monetization bugs** (A2, Q5, S4, S6, T3, P6, O5). Wire
   tested executors to real repos behind access-limits; fix tier-type + fail-open alert; gate
   permissions; make paid results private.
5. **Duplication concentrates risk** (A3, Q4). One `CachedHttpTool` base + one extraction routine.

**Explicitly NOT recommending:** replacing `std::regex` scraping (lexbor already dropped — just
cap response size); C++ writing Postgres directly (sync SQLite→Postgres for beta instead);
building the full 10-command spec / web dashboard / AI Q&A / Stripe (out of MVP scope); further
crypto hardening (already good).

**Definition of done:** (1) live smoke test run, `trade_offers` has ≥1 real row *or* a documented
data-source pivot; (2) CI green-gates merges; (3) zero Critical findings (S1 resolved or
descoped); (4) no unchecked prepare/null-column in `src/store`+`src/cache`, server survives a
malformed frame; (5) HTTP 4xx/5xx never cached; (6) `/price` returns real data behind tier limits,
privately, without crashing on bad input.

### Implementation sketches — top 3

**M0.2 — Live smoke test (highest-value hour).** Source `.env` with a *rotated, throwaway*
credential, run `./build/tibia-listener 2> listener.log`, watch the login state machine and for
`Trade channel joined` (`listener/main.cpp:116`). It will likely hang at the BattlEye handshake
— that's the M1.1 trigger. If it connects, inspect `trade_offers`/`raw_messages` after 10 min.
Gotchas: **violates Tibia ToS / ban risk — disposable account only**; opcodes may be wrong for
the protocol version even if BattlEye passes (capture a packet dump to verify).

**M1.3 — SQLite null/prepare hardening.** Add a `Stmt` RAII wrapper
(`unique_ptr<sqlite3_stmt, finalizer>`) and a `col_text()` helper mapping NULL→`""`; replace the
~10 raw sites in `store/`+`cache/`; check every `prepare_v2`. Write M0.3 characterization tests
first (force a NULL, assert no crash), then refactor to green. Gotcha: columns are `NOT NULL` so
a naive reader "looks fine" — the test must force the NULL.

**M2.1 — `CachedHttpTool` base.** Template parameterized by `{ttl, cache_prefix, parse_fn,
url_builder}` encapsulating validate→key→cache.get→fresh-return→http.get→status-gate→stale-
fallback→parse→cache.put. Migrate the 4 wiki `search_*` tools first. Land QW7 (status gating)
inside the base. Gotchas: TTLs/stale notes differ per tool (pass as params); preserve exact
output strings so tests stay green; replace the fragile `find("Error")` heuristic with the
HTTP-status check.
